# SR-009: Needs-Input Detection and Display

## Status
Draft

## Priority
High

## Dependencies
- SR-008: IPC Notification System

## Description
Detect when agents signal they need input and display this in the UI.

## Constraints

### Detection Method
- Agent sends IPC notification when needs input
- Parse notification for intent
- Update registry accordingly

### Visual Indicators
- Icon in picker list
- Color highlighting
- Sort priority (needs_input first)

## Detection Flow

```
Agent sends IPC → Neovim receives → Handler parses →
Registry updated → Picker refreshed → UI updated
```

## Functions

### `needs_input.set(worktree_id, message, options)`
**Parameters:**
- `worktree_id` (string, required)
- `message` (string, required)
- `options` (table, optional) - Action options

**Returns:** boolean success

**Actions:**
1. Update registry: `needs_input = true`, `needs_input_since = now()`
2. Store pending message and options
3. Trigger picker refresh

### `needs_input.clear(worktree_id)`
Clears needs_input state after user responds.

**Parameters:**
- `worktree_id` (string, required)

**Returns:** boolean success

### `needs_input.get(worktree_id)`
Returns pending input for a worktree.

**Returns:**
```lua
{
  message = "Should I proceed?",
  options = { "Yes", "No" },
  since = "ISO8601",
}
```

## Picker Sorting

When sorting by status, needs_input items should appear first:

```lua
local function sort_entries(entries)
  table.sort(entries, function(a, b)
    -- Needs input first
    if a.needs_input and not b.needs_input then return true end
    if not a.needs_input and b.needs_input then return false end
    -- Then by time waiting (longest first)
    if a.needs_input and b.needs_input then
      return a.needs_input_since < b.needs_input_since
    end
    -- Then by priority (highest first)
    return (a.priority or 99) < (b.priority or 99)
  end)
  return entries
end
```

## User Response Flow

1. User sees notification
2. User opens picker (`<leader>aa`)
3. User selects agent with needs_input
4. User presses `<CR>` to focus agent
5. User responds to agent in terminal
6. Agent sends `input_received` notification
7. System clears needs_input state

## Acceptance Criteria

- [ ] Parse notification content for needs-input signals
- [ ] Update agent status in registry to "needs_input"
- [ ] Display visual indicator in picker (e.g., "⚠" icon)
- [ ] Clear indicator when user addresses the input

## Edge Cases

| Case | Handling |
|------|----------|
| Multiple agents need input | Show all, sort by wait time |
| Input timeout | No automatic timeout, user decides |
| Agent disconnects while needs_input | Keep state, mark as "disconnected" |
| User ignores needs_input | Continue running, eventually timeout conceptually |
