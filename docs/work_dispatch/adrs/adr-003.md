# ADR-003: Worktree Registry Format

## Status
Proposed

## Context
We need to track active worktrees, their agents, bead assignments, and status across Neovim sessions.

## Decision
Store registry as `worktree_root/registry.json` with the following structure:

```json
{
  "version": 1,
  "worktrees": [
    {
      "id": "uuid",
      "path": "/abs/path",
      "branch": "feature-bd-42-add-login",
      "name": "feature-bd-42-add-login",
      "bead_id": "bd-42",
      "bead_title": "Add login functionality",
      "agent": "gemini",
      "session_id": "abc123",
      "status": "running",
      "needs_input": false,
      "needs_input_since": null,
      "created_at": "ISO8601",
      "updated_at": "ISO8601",
      "merge_commit": null,
      "pr_url": null
    }
  ]
}
```

## Alternatives Considered

### Option A: JSON file (CHOSEN)
- Human-readable
- Easy to debug
- Simple implementation
- Pros:
  - Easy to inspect and debug
  - No external dependencies
  - Standard JSON parsing
- Cons:
  - File locking for concurrent access
  - Larger files with many worktrees

### Option B: SQLite database
- Pros:
  - Built-in concurrency
  - Efficient queries
  - Schema migrations
- Cons:
  - External dependency
  - Harder to inspect manually
  - Overkill for simple data

### Option C: Dolt (Beads' database)
- Pros:
  - Version controlled
  - Already available
  - Shareable
- Cons:
  - Complex setup
  - Two database systems

## Schema Design

### Worktree Entry
| Field | Type | Description |
|-------|------|-------------|
| `id` | string (UUID) | Unique identifier |
| `path` | string | Absolute path to worktree |
| `branch` | string | Git branch name |
| `name` | string | Worktree name (directory) |
| `bead_id` | string | Associated bead ID |
| `bead_title` | string | Bead title for display |
| `agent` | string | Agent type |
| `session_id` | string? | Current session ID |
| `status` | string | Current status |
| `needs_input` | boolean | Awaiting user input |
| `needs_input_since` | string? | ISO timestamp |
| `created_at` | string | ISO timestamp |
| `updated_at` | string | ISO timestamp |
| `merge_commit` | string? | Merge commit hash |
| `pr_url` | string? | PR URL after merge |

### Status Values
```lua
local STATUSES = {
  "ready",      -- Worktree created, awaiting agent
  "running",    -- Agent actively working
  "needs_input",-- Awaiting user response
  "paused",     -- Agent stopped
  "done",       -- Work merged
  "rejected",   -- Work rejected
}
```

## File Operations

```lua
local registry = {}

function registry.load()
  local path = worktree_root .. "/registry.json"
  
  local file = io.open(path, "r")
  if not file then
    return { version = 1, worktrees = {} }
  end
  
  local content = file:read("*a")
  file:close()
  
  local data = vim.json.decode(content)
  return data or { version = 1, worktrees = {} }
end

function registry.save(data)
  local path = worktree_root .. "/registry.json"
  
  local file, err = io.open(path, "w")
  if not file then
    return false, err
  end
  
  file:write(vim.json.encode(data))
  file:close()
  
  return true
end

function registry.add(worktree_info)
  local data = registry.load()
  
  local entry = vim.tbl_extend("force", {
    id = vim.fn.uuid_generate(),
    created_at = os.date("!%Y-%m-%dT%H:%M:%SZ"),
    updated_at = os.date("!%Y-%m-%dT%H:%M:%SZ"),
    status = "ready",
    needs_input = false,
  }, worktree_info)
  
  table.insert(data.worktrees, entry)
  registry.save(data)
  
  return entry
end
```

## Version Migrations

```lua
function registry.migrate(data)
  local current_version = 1
  
  if not data.version then
    -- Initial version, no migration needed
    data.version = current_version
  end
  
  if data.version < current_version then
    -- Run migrations
    for v = data.version, current_version - 1 do
      if v == 1 then
        -- No migrations yet
      end
      data.version = v + 1
    end
  end
  
  return data
end
```

## Consequences

### Easier
- Human-readable for debugging
- Easy to query and update
- No external database dependency
- Simple file backup/copy

### Harder
- File locking for concurrent access
- Migration path if schema changes
- May need atomic updates (use temp file + rename)
