# ADR-001: Worktree Storage Location

## Status
Proposed

## Context
Agents need isolated workspaces, but users may want worktrees in a non-standard location (e.g., on a separate drive, in a centralized location across repos).

## Decision
Make the worktree root configurable with `{repo}/.worktree/` as the default. Store the configured path in the module setup.

## Options Considered

### Option A: Fixed `.worktree/` relative to repo (CHOSEN)
- Default: `{repo_root}/.worktree/`
- Pros:
  - Simple, predictable location
  - Easy to find worktrees for a repo
  - Naturally gitignored
- Cons:
  - Tied to repo filesystem
  - Can't share across multiple repos

### Option B: User-configurable path
- Configurable via setup
- Pros:
  - Flexible placement
  - Can use separate drive
  - Centralized location
- Cons:
  - More complexity
  - Cross-repo tracking needed

### Option C: Both (CHOSEN DEFAULT)
- Default is relative, but configurable
- Best of both worlds

## Path Resolution

```lua
local function resolve_worktree_root(configured)
  if not configured then
    -- Default: .worktree/ in repo root
    return vim.fn.fnamemodify(vim.fn.getcwd(), ':p') .. '.worktree/'
  end
  
  if configured:match("^/") then
    -- Absolute path
    return configured
  elseif configured:match("^~") then
    -- Home-relative
    return vim.fn.expand(configured)
  else
    -- Relative to repo root
    return vim.fn.fnamemodify(vim.fn.getcwd(), ':p') .. configured
  end
end
```

## Configuration

```lua
require("work_dispatch").setup({
  worktree_root = ".worktree/",  -- default
  -- or
  worktree_root = "/mnt/fast-disk/agent-worktrees",
  -- or
  worktree_root = "~/.agent-worktrees",
})
```

## Consequences

### Easier
- Users can customize storage location
- Centralized worktree root possible for multi-repo workflows

### Harder
- Must handle path resolution across different locations
- Registry must handle cross-repo references if centralized
- Relative path assumes current working directory context
