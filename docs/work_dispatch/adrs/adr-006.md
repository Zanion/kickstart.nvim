# ADR-006: Integration with Existing Agent Module

## Status
Proposed

## Context
`custom/agent.lua` provides agent spawning and session management. We need to extend it without breaking existing functionality.

## Decision
Create a new module `lua/custom/work_dispatch/` that:
1. Wraps/extends `custom/agent.lua` for worktree-aware spawning
2. Adds worktree creation before agent spawn
3. Injects bead context into the worktree
4. Manages the worktree registry independently

The existing agent picker remains unchanged. The new work dispatch picker is a separate entry point.

## Module Structure

```
lua/custom/
├── agent.lua              # Existing (unchanged)
├── work_dispatch/
│   ├── init.lua           # Main module entry point
│   ├── worktree.lua       # Worktree creation/management
│   ├── registry.lua      # Worktree registry
│   ├── beads.lua          # Beads CLI integration
│   ├── picker.lua         # Telescope picker UI
│   ├── preview.lua        # Terminal preview component
│   ├── actions/
│   │   ├── merge.lua      # Merge action
│   │   ├── reject.lua     # Reject action
│   │   └── reset.lua      # Reset action
│   └── ipc/
│       └── handler.lua    # IPC notification handler
└── plugins/
    └── work_dispatch.lua  # Lazy plugin spec (keybinds, setup)
```

## Integration Points

### Extend Existing Functions

```lua
-- In custom/work_dispatch/agent.lua
local base_agent = require("custom.agent")

local M = vim.deepcopy(base_agent)

function M.spawn_worktree(worktree, agent_name, opts)
  opts = opts or {}
  
  -- Build worktree-aware command
  local cmd = M.build_worktree_cmd(agent_name, worktree)
  
  -- Override cwd to worktree
  opts.cwd = worktree.path
  
  -- Add worktree-specific env
  opts.env = vim.tbl_extend("force", opts.env or {}, {
    NVIM_LISTEN_ADDRESS = worktree.nvim_socket,
    BEAD_ID = worktree.bead_id,
    WORKTREE_ID = worktree.id,
  })
  
  -- Spawn via snacks
  return M.spawn(cmd, opts)
end

function M.build_worktree_cmd(agent_name, worktree)
  local base_cmd = M.agents[agent_name].cmd
  
  -- Add any agent-specific flags
  if agent_name == "gemini" then
    return base_cmd .. " --project-context"
  elseif agent_name == "claude" then
    return base_cmd .. " --project " .. worktree.path
  end
  
  return base_cmd
end

return M
```

```lua
-- In custom/work_dispatch/picker.lua
local base_picker = require("telescope.pickers")

function M.new_worktree_picker()
  -- Reuse telescope infrastructure
```

## Separation of Concerns

| Module | Responsibility |
|--------|----------------|
| `agent.lua` | Spawning terminals, session tracking |
| `work_dispatch/` | Worktree lifecycle, beads integration |
| `agent.lua` keybinds | Direct agent toggle |
| `work_dispatch/` keybinds | Bead dispatch, worktree management |

## Keybind Organization

```lua
-- Existing agent.lua keybinds (unchanged)
vim.keymap.set("n", "<leader>tg", function()
  require("custom.agent").toggle_agent("gemini")
end)

-- New work_dispatch keybinds
vim.keymap.set("n", "<leader>ad", function()
  require("custom.work_dispatch").dispatch_picker()
end)

vim.keymap.set("n", "<leader>aa", function()
  require("custom.work_dispatch").agent_picker()
end)
```

## Configuration

```lua
-- In work_dispatch/init.lua
require("work_dispatch").setup({
  -- Worktree settings
  worktree_root = ".worktree/",
  
  -- Agent settings (extends agent.lua config)
  agents = {
    gemini = {
      extra_flags = "--project-context",
    },
  },
  
  -- Enable/disable integration
  features = {
    auto_context = true,    -- Auto-inject bead context
    ipc_notifications = true, -- IPC-based notifications
  },
})
```

## Backward Compatibility

- `agent.lua` remains unchanged
- Existing keybinds (`<leader>tg`, etc.) continue to work
- New functionality is additive

## Consequences

### Easier
- Separation of concerns
- Can disable work dispatch without breaking basic agent use
- Gradual migration path
- Clear module boundaries

### Harder
- Code duplication risk if not careful
- Two sets of keybinds for agents
- Must keep modules in sync
- Potential confusion about which module to use
