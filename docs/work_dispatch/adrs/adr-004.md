# ADR-004: Agent Status State Machine

## Status
Proposed

## Context
Agents have a lifecycle from spawn to completion. We need a clear state machine to track and display agent status accurately.

## Decision
Implement a simple state machine with the following transitions:

```
[spawned] ──→ [ready] ──→ [running] ──→ [needs_input] ──→ [running]
                              │              │
                              │              │
                          [done] ←────── [rejected]
```

## State Definitions

### `ready`
Worktree created, agent not yet spawned.
- **Enter from:** spawned
- **Exit to:** running (when agent spawned)
- **Display:** ○ (gray circle)

### `running`
Agent actively working in worktree.
- **Enter from:** ready, needs_input
- **Exit to:** needs_input, done, rejected
- **Display:** ● (green circle)

### `needs_input`
Agent awaiting user response.
- **Enter from:** running
- **Exit to:** running (when user responds)
- **Display:** ⚠ (yellow warning)

### `paused`
Agent stopped by user (terminal closed).
- **Enter from:** running, needs_input
- **Exit to:** running (when user resumes)
- **Display:** ⏸ (blue pause)

### `done`
Work merged via PR.
- **Enter from:** running
- **Exit to:** (terminal state)
- **Display:** ✓ (blue check)

### `rejected`
Work rejected by user.
- **Enter from:** running, paused
- **Exit to:** ready (when reset)
- **Display:** ✗ (red cross)

## Transition Table

| From | To | Trigger |
|------|-----|--------|
| - | ready | Worktree created |
| ready | running | Agent spawned |
| running | needs_input | IPC notification |
| needs_input | running | User responds |
| running | done | Merge action |
| running | rejected | Reject action |
| running | paused | Terminal closed |
| paused | running | Focus/resume |
| needs_input | paused | Terminal closed |
| rejected | ready | Reset action |

## Implementation

```lua
local StateMachine = {}

local VALID_TRANSITIONS = {
  ready = { running = true },
  running = {
    needs_input = true,
    done = true,
    rejected = true,
    paused = true,
  },
  needs_input = {
    running = true,
    paused = true,
  },
  paused = { running = true },
  done = {},
  rejected = { ready = true },
}

function StateMachine.can_transition(from, to)
  return VALID_TRANSITIONS[from] and VALID_TRANSITIONS[from][to] or false
end

function StateMachine.transition(entry, to)
  local from = entry.status
  
  if not StateMachine.can_transition(from, to) then
    return nil, string.format(
      "Invalid transition: %s -> %s",
      from, to
    )
  end
  
  local updates = {
    status = to,
    updated_at = os.date("!%Y-%m-%dT%H:%M:%SZ"),
  }
  
  -- Status-specific updates
  if to == "needs_input" then
    updates.needs_input = true
    updates.needs_input_since = updates.updated_at
  elseif to == "running" then
    updates.needs_input = false
    updates.needs_input_since = nil
  end
  
  return updates
end
```

## Sorting Priority

When sorting picker entries:

```lua
local STATUS_PRIORITY = {
  needs_input = 1,  -- Most urgent
  running = 2,
  paused = 3,
  ready = 4,
  rejected = 5,
  done = 6,
}

local function sort_entries(entries)
  table.sort(entries, function(a, b)
    -- Priority first
    local pa = STATUS_PRIORITY[a.status] or 99
    local pb = STATUS_PRIORITY[b.status] or 99
    if pa ~= pb then return pa < pb end
    
    -- Then needs_input wait time
    if a.status == "needs_input" and b.status == "needs_input" then
      return a.needs_input_since < b.needs_input_since
    end
    
    -- Then creation time
    return a.created_at < b.created_at
  end)
end
```

## Consequences

### Easier
- Clear state transitions
- Easy to filter/query by state
- Predictable UI behavior
- Simple status display

### Harder
- State persistence across restarts (handled by registry)
- Handling unexpected terminal close
- Tracking which state changes need IPC notifications
